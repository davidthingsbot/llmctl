# llmctl

Run one or more local LLM servers on a machine and keep everything that talks
to them — agents, a browser chat page, boot config — pointed at the right one.

One command switches the model: the old server is stopped, the new one started
on its own port and health-checked (with rollback on failure), agent configs
are rewritten and their gateways restarted, the web UI is repointed, and the
whole arrangement survives a reboot. Every start is timed and benchmarked into
a stats log that feeds "typical load time" estimates for the next switch.

```
$ llmctl status
models:
  #  name           backend   port   status      description
  1  27b-fp8        vllm      19434  -           Qwen3.6-27B dense FP8, vLLM, 128K, tuned
  2  35b-moe-vllm   vllm      19435  up+healthy  Qwen3.6-35B-A3B MoE FP8, vLLM, 128K

agents:
  hermes   -> qwen3.6-35b-a3b-fp8 @ http://127.0.0.1:19435/v1
  openclaw -> vllm/qwen3.6-35b-a3b-fp8 @ http://127.0.0.1:19435/v1

chat (browser):
  35b-moe-vllm -> http://localhost:8088/ (open-webui, backend http://192.168.1.127:19435/v1)
```

## Design

- **One fixed port per model** (by convention). Models are independent systemd
  user units, so several can run at once if they fit in VRAM — a pre-start
  check verifies free memory per GPU.
- **Backends**: [vLLM](https://github.com/vllm-project/vllm) and
  [llama.cpp](https://github.com/ggml-org/llama.cpp) (`llama-server`), both
  serving the OpenAI-compatible API, key-protected.
- **Machine profile**: host facts (accelerator, GPUs, VRAM, binary paths,
  which agents exist, models directory) live in
  `~/.config/llmctl/machine.conf`, generated by probing the machine. CUDA is
  not assumed — `ACCEL` is `cuda`, `rocm`, or `cpu`.
- **Model registry**: models are small `key=value` files in
  `~/.config/llmctl/models.d/`, created by an interactive wizard (`llmctl add`)
  and hand-editable. (A built-in registry describes the author's dual-RTX-3090
  box; `ENABLE_BUILTIN_MODELS=0` hides it.)
- **Consumers follow the model.** On switch, llmctl rewrites the configs of
  the agents that consume the endpoint and restarts their gateways, and it
  manages an [open-webui](https://github.com/open-webui/open-webui) docker
  container for browser chat with vLLM models (llama.cpp serves its own UI on
  the model port). Each integration is optional per machine.

## Install

```sh
git clone https://github.com/davidthingsbot/llmctl
ln -s "$PWD/llmctl/llmctl" ~/.local/bin/llmctl

llmctl machine init    # probes the host, asks where model weights live,
                       # writes ~/.config/llmctl/machine.conf — review it
llmctl add             # register your first model (interactive wizard)
llmctl set <model>     # make it the active model
```

Requirements: bash, systemd (user units), curl, jq, python3. Optional: PyYAML
(hermes integration), docker (open-webui), nvidia-smi / rocm-smi (GPU checks),
`hf` CLI (model downloads), and a vLLM venv and/or a llama.cpp build.

## Commands

| command | what it does |
|---|---|
| `llmctl` | interactive menu |
| `llmctl <model>` / `llmctl set <model>` | exclusive switch: stop others, start, health-check (rollback on failure), repoint agents + web UI, persist across boot |
| `llmctl up <model>...` | start model(s) alongside what's running (VRAM pre-check) |
| `llmctl down <model>...\|all` | stop + remove from boot |
| `llmctl point <model>` | repoint agents + web UI only |
| `llmctl status` / `list` | units, health, agent targets, chat links, GPU memory |
| `llmctl add` / `remove <model>` | wizard to register a model / unregister it |
| `llmctl bench <model>` | benchmark now: prefill tok/s (via time-to-first-token over a ~1.5K prompt), generation tok/s, GPU util/power sampled in both phases |
| `llmctl stats [model]` | load-time + benchmark history (load, ttft, prefill/gen tok/s) and per-model averages |
| `llmctl env <model>` | print `OPENAI_*` exports for shell clients |
| `llmctl logs <model> [-f]` | the model unit's journal |
| `llmctl machine [init]` | show the effective machine profile / probe + generate it |

`STATS=0` skips the post-start benchmark; `FORCE=1` overrides the VRAM check
and `machine init` overwrite protection.

## Machine profile (`~/.config/llmctl/machine.conf`)

Generated by `llmctl machine init`; every key optional (parsed, not sourced).

```ini
MACHINE_NAME="my-box"
ACCEL=cuda                  # cuda | rocm | cpu (auto-detected when absent)
GPUS_DEFAULT="0,1"          # default visible devices for new models
TENSOR_PARALLEL_DEFAULT=2
VRAM_PER_GPU_MB=24576
MODELS_DIR="/home/me/models"
VLLM_BIN="/home/me/.venvs/vllm/bin/vllm"
VLLM_KEYFILE="/home/me/.config/vllm/api-keys"
LLAMA_SERVER="/home/me/llama.cpp/build/bin/llama-server"
LLAMA_KEYFILE="/home/me/.config/llama.cpp/api-keys"
ENABLE_BUILTIN_MODELS=0     # 1 only on the author's dual-3090 box
WIZARD_BASE_PORT=19438
HERMES_ENABLE=0             # agent integrations, off unless present
OPENCLAW_ENABLE=0
WEBUI_ENABLE=1              # open-webui docker container for vLLM chat
WEBUI_PORT=8088
LEGACY_UNITS=""             # old units to displace when llmctl takes over
```

## Model definition (`~/.config/llmctl/models.d/<name>.conf`)

Written by `llmctl add`; example:

```ini
BACKEND=vllm                # or llamacpp
PORT=19438
MODEL_REF="Qwen/Qwen3.6-27B-FP8"   # HF repo (vllm) or /path/to/model.gguf
MODEL_ID="qwen27b"          # served model name clients use
CTX=128000                  # context window advertised to agents
VRAM_MB=23000               # free MiB needed per GPU before start
TENSOR_PARALLEL=2           # vllm; llamacpp uses SERVER_CTX / STREAMS
GPU_UTIL=0.90
EXTRA_ARGS="--enable-prefix-caching"
```

llmctl generates a run script and systemd unit from this on every start —
edit the `.conf`, not the generated files.
